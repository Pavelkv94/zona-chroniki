## D-000 | Шаг 0 | 2026-07-03
Решение: Скелет монорепо на npm workspaces (`packages/*`), ESM (`"type":"module"`),
TypeScript strict, `moduleResolution: "Bundler"`, запуск через `tsx`/`vitest`
без отдельного build-шага. Пакеты: `@zona/shared`, `@zona/sim`, `@zona/headless`,
`@zona/ui`. Внутренние импорты — по алиасам `@zona/*` (symlink workspaces + `paths`).
Причина: ядру нужен headless-запуск в Node без бандлинга; tsx/vitest исполняют TS
напрямую, поэтому build не нужен до появления UI-сборки.
Альтернативы: pnpm workspaces (отклонено — у пользователя npm); ts-node
(отклонено в пользу tsx — быстрее, ESM из коробки); project references с composite
(отклонено — требует declaration/outDir, лишнее без build-шага).
Затрагивает: package.json, tsconfig.base.json, все packages/*/tsconfig.json.

## D-001 | Шаг 0 | 2026-07-03
Решение: Закон №5 (пакет `sim` не тянет DOM/React/Node) сделан ошибкой компиляции:
в `packages/sim/tsconfig.json` намеренно нет `lib: "DOM"` и `types: []` (без node).
Аналогично `shared`. `ui` получает `lib: DOM` + `jsx: react-jsx`, `headless` —
`types: ["node"]` (единственный пакет с доступом к process/fs).
Причина: превратить конституционный закон из декларации в проверяемый инвариант —
любой запретный импорт падает на `npm run typecheck`.
Альтернативы: eslint-правило no-restricted-imports (отклонено — компилятор строже
и не обходится).
Затрагивает: packages/sim/tsconfig.json, packages/ui/tsconfig.json.

## D-002 | Шаг 0 | 2026-07-03
Решение: `npm run test` = `vitest run --passWithNoTests` (exit 0 при нуле тестов).
Причина: чтобы CI-smoke и локальный прогон были зелёными до появления первых тестов
Фазы 0, а не ложно-красными.
Альтернативы: держать exit 1 (отклонено — ломает будущий CI-gate на пустом наборе).
Затрагивает: package.json → scripts.test.

## D-003 | Фаза 0 | 2026-07-03
Решение: `EntityId`/`EventId`/`LocationId` — branded `number` в `@zona/shared`, без импорта bitecs.
В `@zona/sim` они совместимы с `EntityId` bitecs (тоже number).
Причина: shared/ui/headless видят контракты, не таща ECS-движок; сохраняет закон №5 и слабую связанность.
Альтернативы: реэкспорт `EntityId` из bitecs в shared (отклонено — тянет движок в чистый пакет типов).
Затрагивает: `@zona/shared/src`, все контракты ядра.

## D-004 | Фаза 0 | 2026-07-03
Решение: `Rng.fork(label)` выводит seed ребёнка из `hash(label) ^ rootSeed`, а не из текущего state родителя.
Причина: инвариант детерминизма — добавление вызовов в одной системе не должно ломать последовательности
других (риск C-2).
Альтернативы: fork от текущего state (отклонено — хрупко); один глобальный поток на весь мир
(отклонено — любая перестановка систем ломает историю).
Затрагивает: `core/rng.ts`, все системы (берут `fork(system.name)`).

## D-005 | Фаза 0 | 2026-07-03
Решение: шина — append-only лог с монотонным `EventId`; события тика фиксируются в `endTick`;
реактивные системы читают `at(tick-1)`. Счётчик id (`eventSeq`) сериализуется.
Причина: развязать системы от внутритикового порядка (C-1), сохранить причинные цепочки через
save/load (C-4), закон №6.
Альтернативы: немедленная доставка подписчикам в тот же тик (отклонено — зависимость от порядка систем);
эфемерная очередь без лога (отклонено — ломает летопись/причинность).
Затрагивает: `core/events.ts`, `core/scheduler.ts`, `core/snapshot.ts`.

## D-006 | Фаза 0 | 2026-07-03
Решение: любой замер времени и бюджет 1.6 мс/тик реализуются только в `@zona/headless`;
`@zona/sim` не измеряет время и не принимает решений на его основе.
Причина: разрешить конфликт GDD п.13.4 (бюджет) с законом №8 (запрет Date.now/недетерминизма).
Время не должно влиять на состояние мира.
Альтернативы: адаптивный пропуск систем по времени (отклонено — делает историю зависимой от железа).
Затрагивает: `@zona/headless/cli.ts`, бенчи; `core/scheduler.ts` (без таймеров).

## D-007 | Фаза 0 | 2026-07-03
Решение: не-числовые данные NPC (имена, инвентарь, память, отношения) хранятся в `ResourceStore`,
а не в SoA bitecs; итерация и сериализация — только по отсортированным ключам.
Причина: SoA bitecs хранит числа; закон №8 требует сортировки ключей; отделяет «горячие» числовые
компоненты от «холодных» объектных данных.
Альтернативы: сериализация сырых SoA-массивов целиком (отклонено — захват мусора удалённых eid, C-6).
Затрагивает: `core/world.ts`, `core/snapshot.ts`.

## D-008 | Фаза 0 / задача 0.1 | 2026-07-03
Решение: Проблема «призрака» при переиспользовании eid (риск C-6) закрывается очисткой
ресурсов, а НЕ версионированием eid в bitecs. Введена единственная публичная точка удаления
`destroyEntity(world, eid)` в core/world.ts = низкоуровневый `destroyEcsEntity` (ecs.ts,
внутренний) + `resources.purgeEntity(eid)`. `withVersioning` bitecs не включаем.
Причина: версионирование меняет кодировку eid (id+version в одном числе) и усложнило бы
сериализацию 0.5; purge полностью устраняет наследование «холодных» данных покойника
(имя/инвентарь → нарушение закона №3). Подтверждено QA (инвертированный F-1 тест) и ревью.
Альтернативы: withVersioning (отклонено — сложность кодировки eid); чистить в каждой системе
вручную (отклонено — забудут, единая точка надёжнее).
Затрагивает: core/ecs.ts (destroyEcsEntity внутренний), core/world.ts (destroyEntity, purgeEntity),
core/index.ts (не реэкспортирует низкоуровневый ecs-слой).
ОТКРЫТЫЙ ХВОСТ для 0.5: purge чистит только ResourceStore, но НЕ SoA-компоненты bitecs при
reuse eid. Сериализация 0.5 обязана сохранять/восстанавливать только ЖИВЫЕ сущности и не
захватывать компоненты освобождённых eid.

## D-009 | Фаза 0 / задача 0.2 | 2026-07-03
Решение: RNG, который планировщик кладёт в `SystemCtx.rng`, — это `world.rng.fork(`${system.name}@${tick}`)`,
т.е. метка форка включает НОМЕР ТИКА, а не только имя системы.
Причина: уточнение/исправление к D-004. `fork(label)` детерминирован от `(label, rootSeed)` и НЕ
зависит от состояния родителя — поэтому `fork(system.name)`, вычисляемый заново каждый тик, давал бы
ОДНУ И ТУ ЖЕ последовательность на каждом тике (первый разброс выстрела был бы идентичен из тика в тик).
Включение тика в метку даёт разный поток по тикам при полном детерминизме. Бонус: per-system RNG
становится БЕЗ СОСТОЯНИЯ — поток однозначно определяется `(rootSeed, system.name, tick)`, поэтому 0.5
НЕ обязана сохранять rng-состояние на каждую систему (в отличие от «холодных» форков, что живут дольше тика).
Альтернативы: персистентный форк на систему, живущий между тиками (отклонено — вводит per-system
rng-состояние в снапшот, лишняя поверхность сериализации и рассинхрона); форк меткой только по имени
(отклонено — повтор последовательности каждый тик, скрытый недетерминизм физиологии).
Затрагивает: core/scheduler.ts (сборка SystemCtx), core/system.ts (SystemCtx = { world, bus, rng, tick }),
все будущие системы, использующие ctx.rng.

## D-010 | Фаза 0 / задача 0.2 | 2026-07-03
Решение: (1) Тик АТОМАРЕН. Если система бросает в update, планировщик вызывает `bus.discardTick()`
(сброс внутритикового буфера без коммита) и пробрасывает исключение, НЕ вызывая endTick и НЕ сдвигая
world.tick. Мир остаётся в состоянии «до тика». (2) `register` требует непустое УНИКАЛЬНОЕ имя системы
(держит Set<SystemName>, бросает на дубль/пустое).
Причина: (1) без атомарности упавший+повторённый тик коммитил бы события ДВАЖДЫ в append-only лог —
нарушение законов №8/№3 (событие «из воздуха»). (2) метка форка rng по D-009 = `${name}@${tick}`, поэтому
дубль имени дал бы двум системам ИДЕНТИЧНЫЙ поток случайности (коллизия физиологии) + двойное исполнение.
Побочно: id событий отброшенного тика «сгорают» — пропуски в EventId допустимы (контракт требует
монотонности/уникальности, не непрерывности; восстановление после падения — отдельное исполнение, а не
эталонный детерминированный прогон).
Альтернативы: откат eventSeq при discard (отклонено — лишняя связанность, пропуски безвредны); отложить
атомарность как долг (отклонено ревью-советом — двойной коммит в неизменяемый лог слишком опасен для ядра).
Затрагивает: core/scheduler.ts (try/catch/rethrow, Set имён), core/events.ts (метод discardTick, аддитивно).

## D-011 | Фаза 0 / задача 0.5 | 2026-07-03
Решение: eid при deserialize восстанавливаются ТОЧНО через сериализацию/восстановление аллокатора bitecs
`EntityIndex` verbatim (плоские `dense[]/sparse[]/aliveCount/maxId`, versioning=false по D-008). `ecs.ts`
получает две аддитивные внутренние функции: `exportEntityIndex(ecsWorld)` и `createEcsWorldFromIndex(snap)`
(= `createWorld(restoredIndex)`); `snapshot.ts` работает с индексом как с непрозрачным blob через обёртку
`ecs.ts`, не касаясь bitecs напрямую (слой D-008).
Причина: сохраняет тождество eid И порядок freelist → resume совпадает с непрерывным прогоном побитово
(GDD 5.3). eid'ы, зашитые в ключах ResourceStore и в payload'ах событий append-only лога, остаются
валидными БЕЗ ремапа — core не знает схему доменных payload'ов (закон №6) и не может их переписать.
Закрывает хвост D-008: восстанавливаются только живые eid + аллокатор; SoA-слоты покойников не перечисляются.
Альтернативы: remap old→new (отклонено — не переписать eid внутри событий летописи, C-4/закон №6);
собственный аллокатор eid (отклонено — bitecs уже отдаёт аллокатор как данные); addEntity по порядку с
долгом «дыры» (отклонено — при гибели NPC дыры дают другой порядок freelist на resume → скрытый P0).
Затрагивает: core/ecs.ts (аддитивно), core/snapshot.ts, привязка к bitecs ^0.4.0 (миграция ловится
round-trip-тестом при мажоре).

## D-012 | Фаза 0 / задача 0.5 | 2026-07-03
Решение: `SnapshotJSON = { version:1, seed, tick, rngState, eventSeq, ecsIndex, entities[], resources{},
components{}, eventLog[] }`. `hashSnapshot` — FNV-1a по каноничной строке рекурсивного канонизатора (ключи
объектов сортируются по возрастанию; массивы сохраняют порядок; числа через `Number.toString`;
`NaN/±Infinity/undefined/функции` → throw). `ecsIndex` — авторитет по живым eid; `entities[]` (сорт.) —
производное, сверяется на deserialize. `components:{}` в Фазе 0; форма на будущее —
`{ name: { fields:{f:number[]}, eids:EntityId[] } }` только по живым eid.
Причина: `JSON.stringify` даёт порядок ключей = insertion order → нестабильный хэш (нарушение закона №8).
Альтернативы: хэш поверх `JSON.stringify` (отклонено); `entities[]` как авторитет вместо ecsIndex
(отклонено — ecsIndex кодирует порядок freelist, нужный для resume).
Затрагивает: core/snapshot.ts, @zona/shared (тип SnapshotJSON).

## D-013 | Фаза 0 / задача 0.5 | 2026-07-03
Решение: значения ResourceStore — только plain-JSON-данные; `Map/Set/классы/функции/undefined` запрещены
(Map-подобное хранится как отсортированный массив `[eid,value]`). Сериализация ресурсов — рекурсивная
канонизация (сорт ключей), НЕ голый `JSON.stringify`; канонизатор бросает на запретных значениях с
указанием `(key, eid)`.
Причина: порядок ключей объекта нестабилен между прогонами → ломает хэш (закон №8, C-3); Map/Set-итерация
без сортировки прямо запрещена законом №8. Ранний throw ловит нарушение в тесте/ревью.
Альтернативы: `JSON.stringify` без канонизации (отклонено); per-key кодеки (отклонено для Фазы 0).
Затрагивает: core/snapshot.ts (канонизатор), контракт ResourceStore (значения = plain data), профильные
инженеры (relations/memory как сорт. массивы).

## D-014 | Фаза 0 / задача 0.5 | 2026-07-03
Решение: снапшот сохраняет состояние ТОЛЬКО корневого rng (`rngState` uint32); per-system форки не
сохраняются (stateless, D-009); шина восстанавливается через seam `createEventBus(getTick,{eventSeq,log})`.
Полный набор round-trip: `{ ecsIndex, resources, tick, seed, rngState, eventSeq, eventLog }`. Граничный
инвариант: любой «холодный» форк, живущий дольше одного тика, обязан быть либо пересоздаваемым каждый тик
(метка включает tick, как D-009), либо его `state` добавляется в снапшот. В Фазе 0 таких форков нет.
Причина: минимизировать поверхность сериализации rng; зафиксировать условие достаточности `rngState` корня.
Альтернативы: сохранять per-system rng-состояние (отклонено — D-009); не фиксировать граничное правило
(отклонено — риск скрытого недетерминизма при первом долгоживущем форке).
Затрагивает: core/snapshot.ts, core/rng.ts (restoreRng), core/events.ts (init-seam), будущие системы.

## D-015 | Фаза 0 / задача 0.5 | 2026-07-03
Решение: 0.5 режется на 0.5a (канонизатор + `hashSnapshot` + `serialize` + `exportEntityIndex`) → 0.5b
(`deserialize` + `createEcsWorldFromIndex` + гейт round-trip/resume-детерминизма). Строго последовательно;
форма `SnapshotJSON` замораживается в 0.5a.
Причина: объём не влезает в одну сессию; 0.5a даёт независимо тестируемый артефакт (стабильный хэш).
Альтернативы: одна задача 0.5 (отклонено — превышает сессию).
Затрагивает: docs/ROADMAP.md, процесс core-engineer.

## D-016 | Фаза 0 / задача 0.5b | 2026-07-03
Решение: `createEcsWorldFromIndex` восстанавливает мир в ДВА шага, а не одним `createWorld(index)`.
Шаг 1: `createWorld(revivedIndex)` — bitecs распознаёт объект с `dense/sparse/aliveCount` и кладёт его
как `ctx.entityIndex` verbatim (аллокатор eid+freelist восстановлен; `entityExists`/`spawnEntity` работают).
Шаг 2: ВРУЧНУЮ населяем `ctx.entityComponents` (Map) пустым `Set` на каждый ЖИВОЙ eid (= первые
`aliveCount` элементов `dense`), т.к. `getAllEntities` bitecs 0.4 читает ИМЕННО `entityComponents`, а
`createWorld` оставляет её пустой → без шага 2 `allEntities(restored)` вернул бы `[]` при «живых» eid в
индексе. Заполняем прямой записью в Map, а НЕ через `addEntity`: `addEntity` выдал бы НОВЫЙ eid (recycle
из freelist или `++maxId`) и сдвинул аллокатор, сломав тождество eid и порядок freelist. Касание
`$internal` — только в `ecs.ts` (слой D-008). Плюс: `deserialize` держит GUARD `allEntities(ecs) ===
snap.entities` (авторитет — `ecsIndex`, D-012); при расхождении бросает (битый/подделанный снапшот).
Побочно: `world.ts` получил внутренний seam `createResourceStore()` (регидратация ресурсов в `deserialize`),
`snapshot.ts` — `deserialize` собирает `SimWorld` с той же инициализацией `bus ↔ world.tick`, что
`createSimWorld`, и DEEP-CLONE значений ресурсов (изоляция снапшота от мира). Ни `createEcsWorldFromIndex`,
ни `createResourceStore` не реэкспортируются из `@zona/sim`.
Причина: `getAllEntities` и `entityExists` bitecs 0.4 читают РАЗНЫЕ структуры (`entityComponents` Map против
`entityIndex`); восстановление одного индекса недостаточно для полноценного мира. Прямое население без
`addEntity` — единственный способ сохранить eid+freelist побитово (условие resume, D-011).
Альтернативы: `addEntity` в цикле по живым (отклонено — выдаёт новые eid, ломает freelist/тождество);
патчить bitecs, чтобы `getAllEntities` читал индекс (отклонено — форк движка, риск при апгрейде);
хранить список живых отдельно в снапшоте (отклонено — `ecsIndex` уже авторитет, дублирование рассинхронит).
Проверено гейтом resume-детерминизма (snapshot.deserialize.test.ts): split-прогон через save/load совпадает
с непрерывным по хэшу и логу событий.
Затрагивает: core/ecs.ts (createEcsWorldFromIndex — аддитивно), core/snapshot.ts (deserialize),
core/world.ts (createResourceStore — внутренний seam), core/index.ts (реэкспорт deserialize), привязка
к bitecs ^0.4.0 (структура entityComponents ловится resume-тестом при мажоре).

Правки по ревью 0.5b (та же задача, ужесточение guard'ов read-path):
- GUARD ресурсов (закон №3, симметрия entities-GUARD): `deserialize` строит множество живых eid и при
  регидратации бросает, если пара `[eid,value]` ссылается на НЕ живой eid. Иначе подделанный/битый снапшот
  прорастил бы «призрак», а следующий `spawnEntity` (reuse eid) унаследовал бы его — предмет/имя из воздуха.
- Согласованность аллокатора: `reviveEntityIndex` проверяет не только форму полей, но и инварианты —
  `maxId === dense.length` и обратимость `sparse[dense[i]] === i` для живых `i ∈ [0, aliveCount)`. Type-валидный,
  но рассогласованный blob (кривой sparse / расходящийся maxId) даёт неверный `entityExists`/`spawn`, поэтому
  падаем явно.
- FAIL-FAST по `components`: Фаза 0 их не хранит (`serialize` пишет `{}`); их восстановление НЕ реализовано.
  Непустой `snap.components` → throw, чтобы будущая фаза с реальными SoA-компонентами реализовала загрузку,
  а не потеряла состояние молча (восстановив пустые Set). ОБЯЗАТЕЛЬНЫЙ ХВОСТ для следующей фазы.

## D-026 | Фаза 1 / задача 1.4 | 2026-07-03
Решение (система Movement + детерминированный pathfinding):
1) ЦЕЛЬ движения = `Task.targetLoc` носителя, если у него есть компонент `Task`; иначе/при
   `targetLoc === loc` сущность стоит (`dest === loc`, D-019). Movement НЕ хранит таксономию
   «каким видам задач нужна локация» — для задач без перемещения TaskSelection (1.8) сама
   выставляет `targetLoc = loc`. Так Movement свободен от магии видов задач (закон №7) и не
   пишет `Task` (только читает — чужая зона).
2) Модель транзита без sentinel (D-019): на тике сущность в РОВНО одной ветке — departure
   (`dest = firstStep`, `etaTicks = edgeLen`, publish `move/departed`, БЕЗ списания eta в этот
   тик) ИЛИ decrement/arrival (`etaTicks -= 1`; при `≤0` → `loc = dest`, publish `move/arrived`).
   Одиночный переход = РОВНО `edgeLen` тиков (departure@T → arrival@T+edgeLen). Мультихоп
   эмерджентен: прибыв (`loc = dest`), на СЛЕДУЮЩЕМ тике сущность снова «стоит» и делает
   departure к следующему шагу → между хопами один тик-передышка на узле (контракт B.1).
3) Скорость: `etaTicks = max(MIN_TRAVEL_TICKS, edgeLen)` (balance/movement, закон №7; edgeLen ≥ 1
   ⇒ фактически = edgeLen). Множители ночь/усталость СОЗНАТЕЛЬНО не применяются в 1.4 — связали бы
   Movement с WorldClock/Needs и сломали «ровно edgeLen тиков»; подключит тюнинг позже.
4) Pathfinding — Дейкстра O(V²) на плотных массивах (10 локаций), в отдельном чистом модуле
   `systems/pathfinding.ts` (абстракция `WeightedGraph`, адаптер `MAP_GRAPH` над data). Tie-break
   (закон №8): выбор неулаженного узла обходом по возрастанию id + строгое `<` ⇒ при равной
   дистанции меньший id; релаксация строгим `<` ⇒ предшественник = первое достигшее минимума
   ребро (при равных путях побеждает маршрут через раньше улаженный = меньший id узел). Кратчайший
   путь по `edgeLen` сам не заходит в тупик Саркофаг (degree=1 не лежит на транзитном маршруте),
   поэтому взвешивание danger — забота TaskSelection, не Movement (D-025).
5) Причинность (закон №6, ЗАМКНУТЫЕ цепочки — правка ревью MAJOR-1/MAJOR-2):
   `move/arrived.causedBy` → последний `move/departed` этого eid в логе (departure текущего шага).
   `move/departed.causedBy` различает хопы ноги:
   • ПЕРВЫЙ хоп → самый свежий committed `task/selected` этого eid, У КОТОРОГО
     `payload.targetLoc === текущему Task.targetLoc` (матч по eid И targetLoc — снимает прилипание
     старого события чужой цели; MAJOR-2); нет такого → `null` (не прилепляем чужую цель).
   • ПРОМЕЖУТОЧНЫЙ хоп (departs, только что прибыв в промежуточный узел) → последний `move/arrived`
     этого eid, чтобы цепочка мультихопа не рвалась (MAJOR-1): `task/selected → dep1 → arr1 → dep2
     → arr2 → …`. Различение: совпадающий `task/selected` НОВЕЕ последнего `arrived` (или прибытий
     не было) ⇒ первый хоп; иначе промежуточный.
   ТАЙМИНГ: контракт TaskSelection 1.8 — `task/selected` закоммичен ДО тика departure; финализируется
   при 1.8. Сейчас Movement матчит по (eid,targetLoc) из committed-лога — робастно и без stale
   независимо от точного тика (замена прежней хрупкой привязки к `at(tick-1)`).
   rng НЕ используется (путь детерминирован графом; закон №2 — случайность только для физиологии).
6) В `@zona/shared/events.ts` добавлены члены union `move/departed {eid,from,to}` и
   `move/arrived {eid,at}`. `task/selected` НЕ добавлен (контракт 1.8 — через sim-architect).
7) Инвариант цели (F-2, закон №4): при `firstStep=undefined` (targetLoc невалиден/недостижим)
   сущность стоит молча БЕЗ событий, Movement НЕ падает. Контракт: TaskSelection (1.8) обязана
   давать только валидный достижимый `targetLoc`; молчаливый вечный простой из-за невалидной цели
   ловится world-инвариантом гейта 1.13 («0 idle > N тиков»), а не крашем здесь.
Причина: минимальная, детерминированная система движения, читающая намерение из ECS и общающаяся
через шину с ПОЛНОЙ (замкнутой) причинностью; pathfinding вынесен и покрыт юнит-тестами независимо от ECS.
Альтернативы: departure и следующий hop в один тик (отклонено — контракт B.1 «на следующем тике»);
привязка причины departed к `at(tick-1)` (отклонено ревью — хрупко: null/stale при внутритиковой
невидимости D-005; заменено матчем по (eid,targetLoc) из committed-лога); падать на невалидной цели
(отклонено — латентный idle ловит гейт 1.13, краш в ядре недопустим). Перф чтения лога (сканы O(n))
НЕ трогаем в 1.4 (Movement пока не в прогоне; дешёвый поиск причины — аддендум архитектора перед 1.8);

`sEat = hasFood ? (W.hunger + W.food)·hunger : −∞` (было `W.hunger·hunger + W.food`).
Причина: плоский бонус W.food(0.3) делал EAT победителем argmax даже при hunger≈0 → сытый
сталкер с едой выбирал EAT, а TaskEffects (1.8e) исполнял его ВСЛЕПУЮ и палил весь пищевой
запас по единице/тик без пользы (в сухой локации — весь стек; в водном поселении DRINK частично
перехватывал). Это иррационально («ест сытым») и подрывало пищевую экономику (спирит закона №3 —
необоснованная ПОТЕРЯ ресурса), искажало бы метрики гейта 1.13. Теперь: при hunger≈0 EAT→~0 и
проигрывает fallback (FORAGE/REST/SLEEP) — сытый не ест; при реальном голоде бонус W.food (тоже
масштабированный голодом) поднимает EAT НАД HUNT в типичных условиях — рационально доесть запас,
прежде чем идти на риск охоты. Константы прежние (W.hunger/W.food в balance/utility, закон №7),
доп. порогов не вводили — выбран чистый вариант «оба слагаемых ×hunger».
Проверка (закон №8): детерминизм/resume целы; голдены 481914ae/c5dff40f не сдвинуты (TaskSelection
не в CLI-прогоне). Тесты: task-selection.test.ts — сытый с едой НЕ выбирает EAT (новый регресс),
голодный с едой (даже при дичи рядом) выбирает EAT, EAT==DRINK тай перекалиброван на новую формулу
(hunger=66,thirst=36, бит-точный тай, побеждает EAT). task-effects.test.ts — характеризующие тесты
переедания ИНВЕРТИРОВАны в регресс (запас сытого не горит); выживательный сценарий сохранил вывод
(конечный запас → истощение → рибаунд голода к критическому → падение hp → мотив HUNT), но на
РЕАЛИСТИЧНОМ горизонте: истощение не мгновенное (~1200+ тиков), а не <10 (артефакт переедания снят).
Механика TaskEffects «EAT при hunger=0 списывает единицу» НЕ тронута — фикс в ВЫБОРЕ, не в исполнении.
Затрагивает: systems/task-selection (формула EAT), balance/utility (док W.food), тесты
task-selection/task-effects. Дополняет D-033.

## D-037 | Фаза 1 / задача 1.9 | 2026-07-03
Решение (система Animals — стада, пастьба/бегство, ПРИЧИННОЕ размножение):

   ДЕТЕРМИНИРОВАННАЯ функция: `tick >= phase(herd) && (tick - phase(herd)) % gestationTicks(species)

   хеш id стада (Knuth mult.), КРАТНЫЙ шагу `every:30` и `< gestation` (значит племенной тик всегда
   попадает на due-тик Animals). Это НЕ случайность и НЕ «X% приплод»: фиксированная функция состояния,
   rng на факт рождения НЕ используется вовсе. На племенном тике рождение ТОЛЬКО ЕСЛИ выполнены ПОРОГИ
   СОСТОЯНИЯ: локальная популяция вида в мажоритарной локации стада < reproCap; loc.forage >
   REPRO_FORAGE_MIN; в локации >= REPRO_MIN_HERD_IN_LOC особей стада (родители физически существуют,

   племенной тик через gestation). Максимум ОДНО рождение на стадо за племенной тик. Resume: перепись

## D-039 | Фаза 1 / задача 1.10b | 2026-07-04
Решение (Encounter Resolver + система Encounters, охота человек-vs-животное; реализует D-022):
1) РЕЗОЛВЕР — ЧИСТАЯ функция `resolveEncounter(EncounterInput): EncounterOutcome` над `Combatant[][]`
   (`Combatant = {eid,side,power,ammo,melee,health}`), БЕЗ ECS/шины/данных/глобалей. Вид НЕ хардкодится
   (D-022): зверь = `ammo=0,melee>0,power` из species.json, стрелок = `ammo>0,power=навык×оружие`; две
   стороны из людей идут той же сигнатурой (Фаза 2 бесшовна). Раунды `1..maxRounds` ВНУТРИ функции (резолв
   целиком в одном вызове) — синхронно (снимок живых на начало раунда, урон после обхода) сорт. по (side,eid);
   цель = мин-eid живой враг. Развязка из СОСТОЯНИЯ: сторон с живыми `<=1` → sideWon/mutualBreak; иначе слом
   морали `потери/начало >= MORALE_BREAK_FRACTION` (сломанные бегут); лимит без развязки → stalemate.
2) ЗАКОН №2 (разброс, НЕ «X% шанс убить»). Единственный rng — физический РАЗБРОС выстрела/удара
   (`rng.next()`), попадание = ДЕТЕРМИНИРОВАННОЕ `spread < acc(power, ближний?)`, где `acc` — «конверт
   точности» из силы/навыка и констант balance (BASE/MELEE_ACCURACY + powerNorm×SKILL_ACCURACY_MULT, кламп
   MIN/MAX). Это трактовка «разброс физиологии», прямо санкционированная balance/combat.ts, а не бросок над
   самим фактом смерти. Решение о завязке боя — состояние (человек с HUNT встал в локации с живой дичью), без
   «шанса нападения».
3) КАНАЛ ЗДОРОВЬЯ. У `EncounterOutcome` нет карты hp по eid (форма фиксирована контрактом), поэтому итоговое
   здоровье резолвер ПИШЕТ обратно в ИЗМЕНЯЕМОЕ `Combatant.health` переданных объектов (единственный побочный
   эффект; функция остаётся детерминированной `(вход,rng)→(тот же outcome, те же мутации)`). Encounters держит
   те же объекты по eid и читает health после резолва.
4) СИСТЕМА Encounters (`every:1`, порядок B.1: после Movement/TaskEffects, ДО Animals/Death). Детект: стоящий
   человек (Human,Alive,Health,hp>0) с `Task.kind===HUNT` в локации с живым co-located животным. Цель =
   `Task.targetEid`, если это живое (hp>0, existsEntity+Animal+Alive+co-located, D-029) животное; иначе
   мин-eid живая дичь в его локации. Группировка по (loc,цель): несколько охотников на одну дичь → одна
   сторона side0; один бой на (loc,цель) за тик. Combatant человека: `power=shooting×HUMAN_WEAPON_POWER` при
   совместимом оружии+патронах (иначе 0), `ammo`=патроны калибра оружия из инвентаря, `melee=HUMAN_UNARMED_MELEE`.
5) ПРИЧИННОСТЬ (D-030). `encounter/started.causedBy` = `spottedEvent` цели из contacts мин-eid охотника →
   иначе `Task.causeEvent` (task/selected) → иначе null. `encounter/resolved.causedBy` = id started. Убитым
