# 1. VISION STATEMENT

Это не игра про Зону. Это **симулятор Зоны, внутри которого позже появится игра**.

Мир существует независимо от наблюдателя. Если запустить симуляцию и уйти на несколько часов, при возвращении мир будет другим: одни группировки усилились, другие исчезли, погибли конкретные люди с именами и историями, мутанты мигрировали, экономика перестроилась. Каждая сессия порождает уникальную летопись, которую невозможно повторить.

**Эталонная история, которую должен уметь порождать симулятор:**

> «Я вообще не участвовал в войне. Через месяц одна группировка исчезла из-за голода, потому что зимой волки уничтожили стадо оленей, а торговый караван был разграблен. Когда я пришёл в поселение, там уже жили совсем другие люди.»

**Источники вдохновения (не копирования):** S.T.A.L.K.E.R. A-Life, Космические Рейнджеры, Kenshi, RimWorld, Dwarf Fortress, Shadows of Doubt.

# 2. СТОЛПЫ ДИЗАЙНА (Design Pillars)

Каждое дизайн-решение проверяется по этим четырём столпам. Если решение нарушает хотя бы один — оно отклоняется.

**P1. Мир не знает об игроке.**
Любая система обязана работать при полном отсутствии игрока. Контрольный вопрос перед реализацией любой функции: «Будет ли эта система продолжать работать, если игрок исчезнет из мира?» Если нет — архитектура неверна.

**P2. Причинность вместо случайности.**
Никаких «5% шанс нападения». Вместо этого: бандиты увидели караван → оценили силы → решили напасть. RNG допустим только на уровне первичных «семян» мира (характеры, начальные позиции артефактов) и физиологии (точность выстрела). Каждое событие обязано иметь восстанавливаемую причинную цепочку.

**P3. Всё физически существует.**
Нет абстрактных «+100 еды». Есть конкретная консерва, лежащая в конкретном ящике конкретного поселения. Предмет появляется в мире, только если его кто-то нашёл, создал, добыл, вырастил, перевёз или украл. Нет бесконечных магазинов. Нет спавна из воздуха.

**P4. Радио — сердце мира.**
Наблюдатель узнаёт о мире так же, как его жители — из радиоэфира. Радио передаёт неполную, запаздывающую, иногда ложную картину. Чтение эфира = чтение романа, который пишет симуляция.

---

# 3. ЗАПРЕЩЕНО (Hard Constraints)

Эти вещи не реализуются никогда, ни в каком виде, ни под каким предлогом:

- Сюжетные миссии и квесты (существуют только события)
- NPC, которые ждут игрока или знают о нём что-то сверхъестественное
- Бессмертные персонажи (включая торговцев и лидеров)
- Заскриптованные нападения
- Спавн врагов/предметов возле игрока или где-либо ещё «из воздуха»
- Бесконечные ресурсы и бесконечные деньги у торговцев
- Читерский AI (знание сквозь туман войны, телепортация, чтение мыслей)
- Магические механики
- NPC без имени и фамилии (никаких Stalker_15)
- NPC, стоящие без дела (у каждого всегда есть текущее занятие)
- Исчезновение NPC без причины

Правила одинаковы для игрока и NPC.

# 4. МИР

## 4.1. Структура карты

Мир представлен **графом локаций**: 30–45 узлов, соединённых рёбрами-маршрутами.

Каждый узел (локация) имеет:
- Тип: база фракции, поселение, блокпост, точка интереса (свалка техники, заброшенный завод, деревня, аномальное поле), лес, поле, болото, руины
- Координаты на схематичной карте (для отрисовки и расчёта расстояний)
- Ёмкость укрытий (сколько людей может безопасно ночевать)
- Уровень опасности (производная: кто здесь обитает, сколько трупов, аномалии)
- Ресурсный профиль (что здесь можно добыть: дичь, хабар, дрова, вода)
- Вероятностный профиль спавна артефактов (аномальные поля рождают артефакты после выбросов — см. 4.4)
- Список сущностей, находящихся здесь сейчас

Рёбра (маршруты) имеют: длину (время пути), тип покрытия (дорога/тропа/бездорожье — влияет на скорость), простреливаемость/скрытность (влияет на засады).

**География (по ТЗ):**
- **Юг** — база Военных + 2–3 блокпоста вдоль южной границы
- **Запад** — база Сталкеров + 2–3 поселения-стоянки
- **Восток** — база Долга + 1–2 укреплённых поселения
- **Север** — база Бандитов (ангар), стационарных поселений нет, но есть «сборные точки» на точках интереса
- **Центр** — нейтральная зона: точки интереса, аномальные поля, руины, наибольшая концентрация мутантов

## 4.2. Фракции

### Сталкеры (одиночки)
- **Численность:** ~40–50 человек (самая крупная фракция)
- **Поведение:** ходят по всей карте в поисках артефактов, охотятся на животных и мутантов, торгуют
- **Экономика:** артефакты + части мутантов + шкуры → деньги → снаряжение, еда, патроны
- **Отношения:** нейтралитет ко всем, к бандитам — неприязнь (не агрессия). Личный опыт меняет отношение (см. 6.4)
- **Организация:** слабая; лидер базы — скорее авторитет, чем командир. Ходят поодиночке, парами, редко группами по 3–4

### Военные
- **Численность:** ~25–30 человек
- **Поведение:** сидят на базе и блокпостах, патрулируют к ближним точкам интереса, вглубь карты не заходят
- **Экономика:** снабжение извне — **физические колонны** через Периметр по расписанию (раз в N дней: патроны, еда, медикаменты, топливо). Колонну можно перехватить/уничтожить → у военных начинается реальный дефицит
- **Отношения:** к Долгу нейтрально, к остальным плохо (не агрессивно; агрессия — только при нарушении их периметра или по итогам инцидентов)
- **Организация:** жёсткая иерархия: комендант → офицеры → рядовые. Приказы, смены караулов, расписание патрулей

### Долг
- **Численность:** ~20–25 человек
- **Поведение:** военизированные патрули, зачистки мутантов, охота. Артефакты не собирают (идеология)
- **Экономика:** продают части мутантов и услуги «зачистки», покупают снаряжение. Лучшая экипировка на душу населения
- **Отношения:** нейтралитет ко всем. Ненависть к мутантам — приоритетная цель патрулей
- **Организация:** армейская дисциплина, генерал на базе, отделения по 3–5 бойцов

### Бандиты
- **Численность:** ~20–25 человек
- **Поведение:** грабят одиночек и пары (сталкеров в первую очередь), слушают эфир для наводок, собираются на точках интереса для засад
- **Экономика:** паразитическая — награбленное сбывают своему скупщику; если грабить некого, начинают голодать → вынуждены охотиться или идти на риск (нападение на караваны, поселения)
- **Отношения:** враждебность ситуативная: одиночку грабят, группу из 4+ пропускают (оценка сил, P2)
- **Организация:** атаман + «бригады» по 2–4; внутренняя грызня за добычу возможна

### Нейтральные жители (опционально, фаза 3+)
Механики, торговцы, медики без фракционной принадлежности на базах.

## 4.3. Поселения

Каждое поселение (включая базы) имеет:

| Параметр | Описание |
|---|---|
| Население | Конкретный список NPC (не число!) |
| Запасы | Физические предметы на складах: еда, вода, патроны (по калибрам), медикаменты, стройматериалы, инструменты, топливо |
| Электроэнергия | Генератор + топливо, влияет на: свет ночью (безопасность), работу мастерской, радиус радио |
| Мораль | Производная: сытость, потери, победы, качество лидера |
| Безопасность | Производная: укрепления, число охранников, освещение, история нападений |
| Лидер | Конкретный NPC. Его смерть → кризис преемственности |
| Производство | Что здесь делают: ремонт, готовка, самогон, патроны (кустарные) |
| Потребление | Суточный расход еды/воды/топлива = f(население) |

**Развитие:** при устойчивом профиците ресурсов поселение строит (очередь строительства требует стройматериалы + рабочие руки + время): вышка → склад → медпункт → мастерская → укрепление периметра. При устойчивом дефиците: мораль падает → жители уходят в другие поселения → поселение может быть покинуто.

**Смерть ключевого специалиста меняет мир:** погиб единственный медик → раненые умирают чаще; погиб механик → оружие деградирует; погиб торговец → экономика поселения рушится, караваны туда не ходят.

## 4.4. Артефакты и Выбросы

- Артефакты рождаются **не случайно по карте**, а в аномальных полях (4–6 полей в центральной части)
- **Выброс** — периодическое событие Зоны (раз в 3–7 игровых дней, с предвестниками за час: помехи в эфире, беспокойство животных). Во время выброса: все живые ищут укрытие; кто не успел — погибает или получает урон; после выброса аномальные поля «перезаряжаются» и с некоторой вероятностью рождают новые артефакты
- Артефакт обнаруживается сталкером при проходе рядом (навык «чутьё» + детектор в инвентаре повышают радиус обнаружения)
- Иерархия ценности: «Медуза» (дешёвый) → «Каменный цветок» → «Ночная звезда» → «Вспышка» → «Лунный свет» (редкий, дорогой)
- Выброс — главный «водитель» экономики сталкеров и главный источник смертности вне боёв

## 4.5. Фауна

### Животные
| Вид | Поведение | Опасность |
|---|---|---|
| Олени | Стада 4–8, мигрируют за кормом (сезонность/погода), убегают от всех | Нет |
| Кабаны | Стаи 3–6, территориальны, атакуют при приближении | Средняя |

### Мутанты
| Вид | Поведение | Опасность |
|---|---|---|
| Псевдособаки | Стаи 4–7, охотятся на оленей и одиночек, трусливы при потерях стаи | Средняя |
| Кровосос | Одиночка, ночной охотник, идёт на шум, засадный стиль | Высокая |
| Химера | Одиночка, territorial apex predator, охотится на кабанов и людей | Экстремальная |

### Зомби
- Медленные, опасны вплотную, легко уничтожаются на дистанции
- Бродят, реагируют на шум (выстрелы стягивают зомби с соседних локаций!), сбиваются в стаи
- Могут осадить и уничтожить ослабленное поселение
- Популяция пополняется естественно: погибшие в аномальных зонах люди с некоторой вероятностью поднимаются (лорное обоснование, без спавна из воздуха)

### Экосистема (пищевые цепи)
```
трава/корм ← погода/сезон
   ↓
олени ← волки псевдособаки, охотники-люди
   ↓
кабаны ← химера, охотники
   ↓
псевдособаки ← Долг (зачистки), химера (конкуренция)
```
Животные размножаются (при достатке корма), мигрируют, умирают, конкурируют. Перебили оленей → псевдособаки голодают → начинают нападать на людей и поселения → Долг усиливает патрули → ...

**Дроп:** из мутантов с шансом выпадают части (лапа псевдособаки, щупальца кровососа, коготь химеры) — продаются торговцам (учёные за Периметром платят). Из животных — мясо и шкуры.

## 4.6. Погода и время

- **Цикл дня:** день/ночь. Ночью: люди идут в ближайшее безопасное место, видимость падает, кровососы активизируются, бандиты предпочитают сумерки
- **Погода:** солнце / облачно / дождь / туман / гроза. Влияет на: дальность видимости (засады!), скорость перемещения, слышимость выстрелов, активность животных, урожайность охоты, шанс простудных болезней, помехи радио (гроза)
- **Время:** 1 тик = 1 игровая минута. Скорости наблюдения: пауза / ×1 / ×10 / ×60 / ×600

## 4.7. Приток населения

Новые NPC не рождаются из воздуха — они **приходят в Зону извне**, малыми группами (1–3 человека) через южную границу (мимо блокпостов военных, «дырами» в Периметре).

- **Частота притока = f(привлекательность Зоны).** Привлекательность растёт от событий, «просачивающихся» наружу: найден дорогой артефакт, разбогатевший сталкер ушёл из Зоны, торговец хорошо платит. Падает от волн смертей, эпидемий, разгула бандитов.
- Технически: летопись и слухи имеют «внешнее эхо» — скользящая оценка привлекательности, из которой планировщик формирует группы новичков с генерацией имени, характера и стартового инвентаря (минимального: ПМ, пара консерв, немного денег).
- Новички почти всегда пополняют сталкеров (иногда — бандитов, если привлекательность падает, а приток идёт: в Зону бегут от проблем). Военные и Долг пополняются иначе: военные — ротацией с колоннами снабжения, Долг — вербовкой опытных сталкеров (решение через utility: боевой опыт + отношение к мутантам + харизма вербовщика).
- Это замыкает демографию: смертность ↔ приток, и создаёт петлю обратной связи: богатая Зона → больше новичков → больше добычи бандитам → больше смертей → приток падает → бандиты голодают → ...

# 5. СИМУЛЯЦИЯ: АРХИТЕКТУРА ДВУХ УРОВНЕЙ

Ключевое техническое решение проекта.

## 5.1. Глобальный уровень (Strategic Layer) — всегда работает

Обновляется каждый тик (быстрые системы) или каждые 5–10 тиков (медленные):

| Система | Частота | Что делает |
|---|---|---|
| Needs | каждый тик | голод, жажда, усталость, здоровье всех NPC |
| Movement | каждый тик | перемещение сущностей по рёбрам графа |
| Perception | каждый тик | кто кого видит в пределах локации / соседних локаций |
| Encounters | каждый тик | обнаружение контакта → передача в Encounter Resolver |
| Weather | 10 тиков | смена погоды, день/ночь |
| Economy | 10 тиков | производство, потребление, цены, дефициты |
| Faction AI | 30 тиков | стратегические решения: патрули, караваны, рейды, дипломатия |
| Ecology | 30 тиков | размножение, миграция, голод животных |
| Sickness | 60 тиков | болезни, выздоровление, эпидемии |
| Rumors | 10 тиков | распространение и искажение слухов |
| Chronicle | событийно | запись значимых событий в летопись |

## 5.2. Локальный уровень (Tactical Layer) — MVP: абстрактный резолвер

В MVP боестолкновения разрешаются **абстрактным резолвером** (см. 7), покадровая тактика не симулируется. В фазе рендера (Фаза 4+) активная локация может «разворачиваться» в детальную симуляцию: A*-поиск пути, укрытия, прицеливание, позиционный бой.

Принцип as in STALKER A-Life: сущности в offline-режиме живут «статистически» (позиция, цель, состояние), в online-режиме — детально. Переход бесшовный и не меняет судьбу сущности (offline-бой и online-бой должны давать статистически схожие исходы — это тестируется).

## 5.3. Детерминизм

- Вся симуляция работает на **seeded PRNG** (mulberry32/sfc32). Один seed → одна и та же история мира
- Это критично для: отладки («перемотай к дню 21 и посмотри, почему погиб Жуков»), тестов, балансировки
- Логика полностью отделена от рендера: симуляция может работать headless в Node (для тестов и балансовых прогонов) и в Web Worker (в браузере, чтобы UI не блокировался)

# 6. NPC: АНАТОМИЯ ЖИТЕЛЯ ЗОНЫ

## 6.1. Атрибуты

Каждый NPC — полноценная запись:

```typescript
interface NPC {
  id: EntityId;
  name: { first: string; last: string; nickname?: string }; // Алексей Жуков ("Лис")
  age: number;
  sex: 'm' | 'f';
  faction: FactionId;
  traits: Trait[];            // характер: трус/храбрец, жадный/щедрый, мстительный, болтливый...
  profession: Profession;
  // Состояние
  health: number;             // 0..100, части тела — в фазе 4+
  hunger: number; thirst: number; fatigue: number; fear: number;
  sickness: Sickness | null;
  combatExperience: number;   // растёт от боёв, влияет на резолвер
  // Социальное
  relations: Map<EntityId, Relation>;   // -100..+100 + теги причин
  reputation: Map<FactionId, number>;   // как фракции к нему относятся
  money: number;
  // Материальное
  inventory: Item[];          // физические предметы, есть вес и объём
  home: LocationId | null;
  workplace: LocationId | null;
  // Разум
  currentTask: Task;          // всегда есть! (P-запрет на "стоять без дела")
  shortTermGoal: Goal;
  longTermGoal: Goal;         // "накопить на экзоскелет", "отомстить за брата", "уйти из Зоны"
  memory: MemoryRecord[];
  skills: { shooting: number; stealth: number; medicine: number;
            repair: number; trading: number; survival: number; artifactSense: number };
}
```

**Генерация имён:** пул славянских имён/фамилий + генератор кличек (по чертам характера и профессии: снайпер-молчун → «Глухарь»). Имена уникальны в рамках мира.

## 6.2. Принятие решений: Utility AI + иерархия целей

NPC выбирает текущую задачу через **utility scoring** — у каждого возможного действия считается полезность из потребностей, черт, целей и обстановки:

```
score(охота)   = голод×0.4 + запасДичиРядом×0.3 + навыкВыживания×0.2 − страх×0.3 − ночь×0.4
score(поход за артефактами) = жадность×0.3 + слухОбАртефакте×0.5 + долгосрочнаяЦель×0.2 − опасностьМаршрута×0.4
score(сон)     = усталость×0.8 + ночь×0.5 + безопасностьМеста×0.3
```

Иерархия: **долгосрочная цель** (недели) → порождает **краткосрочные цели** (день) → порождают **задачи** (минуты-часы). Пример: «накопить 5000 на бронежилет» → «сходить к Свалке за артефактами» → «дойти до Свалки» → «идти по ребру A-B».

Черты характера — это модификаторы весов: трус имеет ×2 к весу страха, жадный ×1.5 к весу денег, мстительный не забывает обиды (замедленный decay памяти о вреде).

## 6.3. Память

```typescript
interface MemoryRecord {
  tick: number;                 // когда
  type: 'helped' | 'robbed' | 'killed' | 'met' | 'sawDanger'
      | 'foundResource' | 'comradeDied' | 'heardRumor';
  subject: EntityId | null;     // кто
  location: LocationId;         // где
  strength: number;             // 0..1, затухает со временем
  isFirsthand: boolean;         // видел сам или услышал по радио
}
```

- Затухание: strength -= decayRate × Δt. Мелкое (встретил) забывается за дни, крупное (убили товарища) — за месяцы, у мстительных — почти никогда
- Память влияет на: отношение (6.4), выбор маршрутов (там видел химеру → обходит), слухи (пересказывает по радио)
- Память из вторых рук (isFirsthand=false) слабее и может быть ложной

## 6.4. Отношения и репутация — двухслойная система

**Слой 1 — личные отношения (NPC ↔ NPC):** число −100..+100 + причины. Меняются от прямого опыта: помог в бою +20, поделился едой +10, ограбил −40, убил товарища −80.

**Слой 2 — фракционная репутация (NPC ↔ Faction):** базовая матрица фракций (военные↔сталкеры: −20, все↔бандиты: −40, и т.д.) + личная поправка.

**Репутация живёт:** бандит спас сталкера → тот сталкер лично относится к нему +30 → рассказывает по радио → часть сталкеров улучшает отношение к этому бандиту, а часть бандитов считает его предателем (−20 внутри своей фракции). Итог: возможны перебежчики, изгнанники, личные вендетты — всё это эмерджентно.

## 6.5. Распорядок дня

Через utility это возникает само: утром (усталость низкая, голод средний) → завтрак → работа/вылазка; вечером (усталость высокая) → возврат к ближайшему безопасному месту → ужин → сон. Профессия задаёт «рабочие» задачи: охранник — смены караула, торговец — торговля и учёт, медик — лечение, механик — ремонт очереди оружия, повар — готовка из сырья.

## 6.6. Профессии

Сталкер-добытчик, Охотник, Проводник, Медик, Механик, Торговец, Охранник, Разведчик, Повар, Носильщик, Радист, Лидер. Профессия = набор доступных «рабочих» задач + стартовые навыки. NPC может сменить профессию (охранник, оставшийся без поселения, становится сталкером).

# 7. БОЕВАЯ СИСТЕМА (MVP: Encounter Resolver)

Столкновение = вход двух враждебных групп в зону видимости друг друга.

**Шаг 1 — Обнаружение:** f(дальность видимости[погода, ночь], навык скрытности, рельеф ребра/локации). Кто увидел первым — имеет инициативу.

**Шаг 2 — Оценка (обе стороны):** сила = Σ(оружие × патроны × навык × опыт × здоровье) × мораль × укрытия. Каждая сторона выбирает: атаковать / отступить / избежать / переговоры (бандиты: «снимай рюкзак») / засада.

**Шаг 3 — Разрешение боя раундами (1 раунд = 1 тик):** каждый раунд считаются попадания (навык, дистанция, укрытие, погода), тратятся **реальные патроны из инвентарей**, наносятся ранения, проверяется мораль (потери → паника → бегство). Бой слышен: шум распространяется на соседние локации → зомби стягиваются, кровосос идёт на звук, патруль Долга может вмешаться.

**Шаг 4 — Последствия:** трупы остаются в локации **с инвентарём** (лут для нашедшего, источник сообщений «нашёл тело Лиса»), раненые нуждаются в медике, выжившие пишут память, в эфир идут сообщения, событие оценивается для летописи.

Резолвер един для всех: люди vs люди, люди vs мутанты, мутанты vs животные, зомби vs поселение.

# 8. РАДИО — НАРРАТИВНОЕ СЕРДЦЕ

## 8.1. Механика

- У каждого человека — рация. У поселений — стационарные станции (радиус больше, зависит от электричества)
- Сообщение рождается из **события + наблюдателя**: NPC видит нечто значимое → генерируется сообщение от его имени
- **Зона покрытия:** сообщение слышат те, кто в радиусе. Ретрансляция: услышавший может передать дальше (болтливость — черта характера)
- **Задержка и драматургия:** сообщения о бое идут в реальном темпе боя:

```
[14:02] Лис: Контакт! Трое у Свалки!
[14:02] Лис: Штурм!
[14:03] Лис: Помогите!!
[14:03] — тишина —
[14:41] Ворон: Нашёл тело Лиса у Свалки. Суки.
[14:44] Череп: Хорошим хабаром поделился твой Лис, хе-хе.
```

- **Помехи:** гроза и выброс глушат эфир (сообщения теряются или приходят обрывками)

## 8.2. Слухи и недостоверность

Сообщение — это **утверждение о мире, а не факт**:
- Ошибка наблюдения: кабан в тумане → «вижу химеру!» → сталкеры сутки обходят локацию, где химеры нет
- Искажение при ретрансляции: «двое бандитов» → «отряд бандитов» → «банда штурмует Свалку»
- Намеренная ложь: бандит даёт ложную наводку на «жирный хабар» → засада
- NPC оценивают доверие к источнику (репутация + личный опыт: этот уже дважды «видел химеру»)

## 8.3. Шаблоны сообщений

Генерация: `шаблон(тип события) + параметры(имена, место, детали) + окраска(характер говорящего)`. Паникёр и ветеран сообщают об одном бое разными словами. Пул из 15–25 шаблонов на тип события, чтобы эфир не был роботизированным.

# 9. ЭКОНОМИКА И ЛОГИСТИКА

## 9.1. Замкнутые контуры

Деньги и предметы не создаются из воздуха. Контуры:

```
Артефактный: выброс → артефакты → сталкер рискует → торговец покупает →
  торговец продаёт учёным за Периметр (единственный "экспорт", приносящий деньги в Зону)
Охотничий: животные → мясо/шкуры → еда поселений / продажа
Мутантный: части мутантов → торговец → за Периметр
Военный: колонны снабжения из-за Периметра (расписание, физический маршрут)
Бандитский: грабёж → скупщик → перераспределение
Производственный: сырьё + мастерская + механик → ремонт/патроны кустарные
```

## 9.2. Торговцы

- Торговец — смертный NPC с **конечными** деньгами и складом
- Цены динамические: дефицит патронов в поселении → патроны дорожают → сталкерам выгодно везти патроны туда → караван → бандиты слушают эфир...
- Караваны — физические группы с грузом и охраной, идущие по рёбрам графа

## 9.3. Дефицит как двигатель драмы

Уничтожена колонна военных → дефицит патронов на базе → патрули урезаны → мутанты плодятся у южной границы → блокпост осаждён. Голод у бандитов → отчаянные рейды → или распад банды. Экономика — главный генератор конфликтов, война — продолжение дефицита.

# 10. СОБЫТИЯ И ЛЕТОПИСЬ

## 10.1. События

События не «выпадают» — они **являются следствиями**. Каждое событие хранит ссылку на причину (`causedBy: EventId`), образуя причинные цепочки, которые можно раскрутить назад:

```
Летопись, День 21: «Поселение новичков покинуто»
  ← осада зомби (день 19-21)
  ← зомби стянулись на шум большого боя (день 19)
  ← бой: рейд бандитов на поселение (день 19)
  ← голод бандитов (дни 14-19)
  ← караван сталкеров, который бандиты «пасли», ушёл другим маршрутом (день 14)
  ← сталкеры услышали по радио о засаде (день 13)
```

## 10.2. Летопись мира (Chronicle)

- Каждое событие получает **оценку значимости** (смерть NPC: вес = его известность; уничтожение поселения: максимум; найден редкий артефакт: средне)
- События выше порога попадают в летопись: `День 3: Сергей Лисенко погиб в засаде бандитов у Свалки`
- Летопись — отдельная вкладка UI + экспорт в текст (шэринг «истории моего мира» — вирусный потенциал)
- Летопись влияет на мир: значимые события порождают слухи и долгую память («у Свалки гиблое место, там Лисенко положили»)

# 11. ИНТЕРФЕЙС НАБЛЮДАТЕЛЯ (MVP)

Формат MVP: **гибрид — схематичная карта + радио-лог**.

```
┌─────────────────────────────────┬──────────────────────────┐
│         КАРТА (Canvas/Pixi)     │   РАДИОЭФИР (React)      │
│  • узлы-локации (иконка+имя)    │  [14:02] Лис: Контакт!   │
│  • точки-сущности с цветом      │  [14:03] Лис: Помогите!  │
│    фракции, кластеры стай       │  ...автоскролл, фильтры  │
│  • погода/время/день N          │  по фракции и типу       │
│  • клик по сущности → карточка  ├──────────────────────────┤
│    (имя, hp, задача, цель,      │   ЛЕТОПИСЬ / ИНСПЕКТОР   │
│     инвентарь, память, связи)   │   вкладки                │
├─────────────────────────────────┴──────────────────────────┤
│  ⏸ ×1 ×10 ×60 ×600   | День 21, 14:03, дождь | стата мира │
└────────────────────────────────────────────────────────────┘
```

Ключевой инструмент — **инспектор сущности**: кликнув на любого NPC, наблюдатель видит его полное состояние, включая память и отношения. Это же главный инструмент отладки.

Пиксель-арт рендер (PixiJS, тайловая карта, спрайты, локальный уровень боя) — Фаза 4, не раньше. Правило из ТЗ: **если симуляция интересна в текстовом логе — проект развивается правильно**.

# 12. ИГРОК (Фаза 5+, за пределами MVP)

Кратко фиксируем принципы на будущее:
- Игрок выбирает фракцию и получает **обычного NPC** под управление (или начинает одиночкой)
- Все правила общие: голод, болезни, смерть (permadeath — мир продолжается, можно взять другого персонажа или наблюдать)

# 13. ТЕХНИЧЕСКИЙ СТЕК И АРХИТЕКТУРА

## 13.1. Стек

| Слой | Технология | Обоснование |
|---|---|---|
| Язык | TypeScript (strict) | типобезопасность контрактов между системами |
| ECS | **bitecs** | производительность (SoA), 150+ сущностей с запасом |
| Симуляция | чистый TS, headless | работает в Node (тесты/балансировка) и Web Worker (браузер) |
| Рендер карты | Canvas 2D (MVP) → PixiJS v8 (Фаза 4) | для схематичной карты Pixi избыточен |
| UI | React 18 + zustand | радио-лог, летопись, инспектор |
| Мост Sim↔UI | Web Worker + postMessage (снапшоты дельт) | UI никогда не блокирует симуляцию |
| PRNG | mulberry32 (seeded) | детерминизм |
| Сборка | Vite | скорость |
| Тесты | Vitest | юнит + симуляционные прогоны |
| Сохранения | IndexedDB (снапшот JSON + seed + tick) | браузер |
| Карта мира | JSON-описание графа (позже Tiled для тайлов) | простота правок |
| Звук | Howler (Фаза 4+) | радио-шумы, эмбиент |

## 13.2. Структура проекта

```
/packages
  /sim              ← ядро, НОЛЬ зависимостей от DOM
    /core           ← ECS-мир, планировщик тиков, PRNG, шина событий
    /components     ← данные (Position, Needs, Memory, Inventory...)
    /systems        ← логика (NeedsSystem, EncounterSystem, RadioSystem...)
    /data           ← JSON: карта, фракции, предметы, имена, шаблоны сообщений
    /balance        ← все числовые константы В ОДНОМ МЕСТЕ
  /ui               ← React-приложение
  /headless         ← CLI: прогон N дней, вывод метрик и летописи
  /shared           ← типы и контракты Sim↔UI
```

**Правила архитектуры:**
1. Системы общаются только через компоненты и шину событий — никаких прямых вызовов SystemA → SystemB
2. Каждая система обязана взаимодействовать минимум с тремя другими (через данные)
3. Каждая новая система = объяснение + диаграмма зависимостей + пример + тесты
4. Все балансовые числа — в `/balance`, никаких магических констант в коде систем
5. Шина событий — единственный источник для: радио, летописи, памяти NPC

## 13.3. Событийная шина (ключевой контракт)

```typescript
type SimEvent =
  | { type: 'entityDied'; who: EntityId; cause: DeathCause; where: LocationId; causedBy?: EventId }
  | { type: 'combatStarted'; parties: EntityId[][]; where: LocationId; causedBy?: EventId }
  | { type: 'artifactFound'; who: EntityId; artifact: ItemId; where: LocationId }
  | { type: 'caravanDeparted'; from: LocationId; to: LocationId; members: EntityId[] }
  | { type: 'settlementAbandoned'; which: LocationId; causedBy?: EventId }
  | ... // ~30 типов
```

Каждое событие имеет id и опциональный causedBy → причинный граф для летописи и отладки бесплатно.

## 13.4. Бюджет производительности

Цель: 150 сущностей-людей + ~100 существ, скорость ×600 (10 игровых часов/сек) при 60 FPS UI. При тике 1 мин это 600 тиков/сек ≈ 1.6 мс на тик. bitecs + системы с頻отами частот (5.1) укладываются с запасом. Perception — самая дорогая система: считать только внутри локаций и по соседним рёбрам, никакого глобального n².

# 15. ROADMAP (фазы с критериями готовности)

Календарных сроков нет — есть **Definition of Done** для каждой фазы. Фаза не начинается, пока не закрыта предыдущая.

## Фаза 0 — Скелет (ядро без контента)
ECS-мир, планировщик тиков с частотами, seeded PRNG, шина событий, headless-CLI, конвейер снапшотов в Worker.
**DoD:** headless-прогон 1000 тиков детерминирован (два прогона с одним seed → идентичный лог событий), покрыт тестами.

## Фаза 1 — Живые тела
Граф карты (10 локаций), 20 NPC одной фракции: потребности, движение, utility-выбор задач (сон/еда/охота/возврат домой), день/ночь, животные (олени, кабаны), простая охота через резолвер.
**DoD:** прогон 10 дней: никто не «стоит без дела», NPC выживают за счёт охоты, смерти от голода объяснимы причинной цепочкой.

## Фаза 2 — Общество
4 фракции, поселения с запасами и потреблением, торговцы, деньги, артефакты + выбросы, encounter resolver (люди vs люди, оценка сил, грабёж), память и отношения, профессии.
**DoD:** прогон 30 дней: бандиты грабят одиночек, но избегают групп; экономика не уходит в бесконечность и не коллапсирует за 30 дней; смерть медика измеримо повышает смертность раненых.

## Фаза 3 — Голос мира
Радио (покрытие, задержки, ретрансляция), слухи с искажениями, летопись с оценкой значимости, мутанты и зомби, шум боя стягивает угрозы, миграция животных, болезни, колонны военных, дипломатия фракций (союзы/предательства), развитие и гибель поселений.
**DoD (главный тест проекта):** прогон 100 дней без наблюдателя порождает ≥3 связные «истории» уровня эталонной (п.1), читаемые из летописи; радио-лог интересно читать 15 минут подряд.

## Фаза 4 — Лицо мира
UI-гибрид (карта + эфир + инспектор + летопись), затем пиксель-арт: тайлы, спрайты, локальный детальный уровень для активной локации, звук.
**DoD:** наблюдатель без знания внутренностей понимает происходящее; offline/online-бои статистически эквивалентны.

## Фаза 5 — Игрок
Управляемый персонаж по общим правилам. Отдельный дизайн-док по итогам наблюдений за симуляцией.

## Метрики «здоровья» симуляции (гоняются headless-CLI на каждой фазе)
- Выживаемость: % NPC, доживших до дня 30/100 (цель: 40–70% к дню 100 — смертность есть, но мир не вымирает)
- Денежная масса и запасы еды: без экспоненциального роста/коллапса
- Активность: 0 NPC в состоянии idle > 5 минут
- Разнообразие: ≥15 различных типов событий в летописи за 100 дней
- Каузальность: 100% событий летописи имеют полную цепочку causedBy до корня

# 16. РАЗРАБОТКА ЧЕРЕЗ ИИ-АГЕНТОВ

## 16.1. Организация работы

Среда: **Claude Code + Cursor**. Один и тот же набор промптов используется в обеих:

- **Claude Code:** общая «конституция» → `CLAUDE.md` в корне репо; агенты → `.claude/agents/<имя>.md` (субагенты с frontmatter `name`, `description`, `tools`)
- **Cursor:** конституция → `.cursor/rules/000-constitution.mdc` (alwaysApply: true); агенты → отдельные rules-файлы, подключаемые вручную под задачу, либо кастомные режимы

**Процесс на одну задачу:** Архитектор (декомпозиция + контракт) → профильный инженер (реализация + тесты) → QA (симуляционные тесты) → Ревьюер (соответствие конституции) → Баланс-аналитик (прогон 30–100 дней, если задето поведение мира). Человек (Pavel) — продакт и финальный ревью.

## 16.2. CLAUDE.md — конституция проекта (общая для всех агентов)

```markdown
# Проект «Зона: Хроники» — симулятор A-Life

Ты работаешь над симулятором живого мира в сеттинге зоны отчуждения.
TypeScript strict, ECS (bitecs), монорепо: /packages/sim (headless ядро),
/packages/ui (React), /packages/headless (CLI), /packages/shared (типы).

## Нерушимые законы (нарушение = задача провалена)
1. ГЛАВНЫЙ ТЕСТ: перед реализацией любой функции ответь на вопрос
   «Будет ли эта система работать, если игрок исчезнет из мира?»
   Если нет — остановись и предложи другую архитектуру.
2. Причинность вместо случайности. Запрещены проверки вида «X% шанс события».
   События возникают из состояния мира. RNG — только seeded PRNG из core/rng.ts
   и только для физиологии (разброс выстрела) и генерации мира.
3. Всё физически существует. Запрещено создавать предметы/деньги/NPC из воздуха.
   Любое появление предмета обязано иметь источник (найден/создан/добыт/украден).
4. Никаких квестов, скриптовых сцен, бессмертных NPC, спавна возле кого-либо,
   NPC без имени-фамилии, NPC в состоянии idle.
5. /packages/sim не импортирует ничего из DOM/React. Симуляция обязана
   запускаться headless в Node.
6. Системы не вызывают друг друга напрямую. Общение — только через компоненты
   ECS и шину событий (core/events.ts). Каждое событие имеет causedBy.
7. Все балансовые константы — только в /sim/balance. Магические числа в
   системах запрещены.
8. Детерминизм: одинаковый seed → одинаковая история. Запрещены Math.random,
   Date.now, итерация по Map/Set без сортировки ключей внутри симуляции.
9. Каждый новый модуль сопровождается: docblock-объяснением, диаграммой
   зависимостей (mermaid в /docs), примером использования, юнит-тестами.
10. Весь контент (фракции, имена, предметы, шаблоны радио) — в /sim/data JSON.
    Код оперирует абстрактными id.

## Definition of Done для любой задачи
- npm run test проходит; добавлены тесты на новое поведение
- npm run sim:100days проходит без ошибок и деградации метрик здоровья мира
- Нет нарушений законов 1–10
```

# 17. ПРИНЯТЫЕ РЕШЕНИЯ (v1.1)

Вопросы, открытые в v1.0, закрыты решениями продакта:
